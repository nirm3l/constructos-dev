# ConstructOS Implementation Report (2026-03-28)

## Scope
This report summarizes what was implemented in this cycle, what is now different in backend/runtime behavior, and what changed in the UI.

It includes:
- Team Mode architecture refactor toward shared runtime context patterns.
- Runner scheduling/queueing improvements.
- Project-level runtime snapshot and visibility upgrades.
- Deploy validation and recreate-from-zero execution outcome.

## What Was Implemented

### 1. Team Mode shared runtime context (core architectural change)
New modules:
- `app/plugins/team_mode/runtime_context.py`
- `app/plugins/team_mode/runtime_snapshot.py`

Key changes:
- Introduced `TeamModeProjectRuntimeContext` as a single source of truth for:
  - Team Mode plugin config loading.
  - Team agent normalization.
  - Member role map.
  - Task list/state/task-like caches.
  - Workflow role derivation.
  - Project runtime activity detection.
  - Active agent load calculation.
- Added `build_team_mode_runtime_snapshot(...)` to produce a deterministic, task-first runtime snapshot for project checks/UX.

Why this matters:
- Removes repeated ad hoc config parsing/state reconstruction across features.
- Reduces divergence between kickoff, task handlers, runner orchestration, and checks APIs.
- Improves consistency and lowers risk of Team Mode behavior drift.

### 2. Task command handlers refactor to runtime context
Updated file:
- `app/features/tasks/command_handlers.py`

Key changes:
- Replaced repeated direct `ProjectPluginConfig` + JSON parsing blocks with `TeamModeProjectRuntimeContext`.
- Centralized reads for:
  - Team Mode enablement.
  - Team agents.
  - Status semantics.
  - Member role mapping.
  - Project runtime activity checks.
- Kept core task domain flow intact (no rewrite of core task model or event model).

Why this matters:
- Team Mode logic is now easier to reason about and maintain.
- Lower chance of inconsistent behavior between different command paths.

### 3. Runner orchestration and queue/scanning improvements
Updated file:
- `app/features/agents/runner.py`

Key changes:
- Added `_ordered_task_ids_for_runner_scan(...)` and reused it in:
  - `run_queued_automation_once`
  - `recover_stale_running_automation_once`
- Improved trigger/schedule queueing:
  - Better event-priority ordering.
  - Reduced repeated DB loading.
  - Workspace selector cache now uses projected columns instead of full row objects.
  - Scheduled task queueing prefetches event-referenced tasks (reduces N+1 reads).
- Failure handling cleanup:
  - Added local `refresh_state()` helper in `_record_automation_failure`.

Why this matters:
- Better determinism in what gets processed first.
- More efficient scans under load.
- Cleaner recovery/failure paths.

### 4. Team Mode kickoff and role handling improvements
Updated files:
- `app/plugins/team_mode/api_kickoff.py`
- `app/plugins/team_mode/task_roles.py`

Key changes:
- Kickoff path now relies on runtime context for team/task state, role resolution, and dispatch readiness.
- Role handling normalized through canonical role mapping.
- Agent selection uses load-aware role picker logic.
- Team Mode labels cleanup ensures legacy `tm.agent:*` noise is removed from label persistence path.

Why this matters:
- Kickoff and dispatch decisions are more consistent and less heuristic.
- Cleaner role/slot semantics for Team Mode routing.

### 5. Project checks API now includes Team Mode runtime snapshot
Updated file:
- `app/features/projects/api.py`

Key changes:
- `project_checks_verify` payload now includes `team_mode_runtime` generated by `build_team_mode_runtime_snapshot(...)`.
- Runtime check payload now surfaces:
  - Agents and their busy/idle state.
  - Task runtime classification (active/runnable/blocked/waiting/etc.).
  - Dispatch and kickoff candidate summaries.

Why this matters:
- UI can show actionable Team Mode orchestration state instead of only static pass/fail checks.

### 6. Startup/deploy blocker fix
Updated file:
- `app/shared/bootstrap.py`

Key change:
- Added missing import for `NOTE_EVENT_UPDATED` to fix startup `NameError` during seeding.

Why this matters:
- This was the concrete blocker preventing stable runtime startup after deploy.

## What Is Different Now (Behavior)

### Team Mode behavior differences
- Team Mode decisions now consistently read from one runtime context abstraction.
- Runtime snapshots are task-first and orchestration-focused (dispatch/kickoff/runnable/blocked state).
- Queue/recovery paths prioritize recent relevant events with deterministic fallback ordering.
- Automation scanning is more efficient and less repetitive in DB access patterns.

## What Changed In The UI

Updated files:
- `app/frontend/src/components/projects/ProjectsInlineEditor.tsx`
- `app/frontend/src/types.ts`

User-visible UI changes:
- Project editor now renders a **Team Mode Runtime** panel (when runtime snapshot is active), including:
  - Parallel limit.
  - Team task counts (active/runnable/blocked/missing instruction).
  - Agent status badges (busy/idle + load count).
  - Top runtime task rows with role/runtime/automation/status/slot/context markers.
- Team Mode/Checks views now include a **Task-First Runtime Snapshot** summary:
  - Dispatch queue size.
  - Kickoff target count.
  - Dispatch/kickoff blocker reasons.
  - Per-task runtime metadata and quick links to task detail.
- Frontend type contracts were extended to support `team_mode_runtime` payload fields used by the new UI blocks.

Net UX effect:
- Team Mode state is now observable from project checks/editor UX without digging through task internals.
- Easier to identify why work is blocked and what will dispatch next.
